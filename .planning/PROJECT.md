# UE Incremental Trust Programming System（UE-ITPS）

## What This Is

UE-ITPS 是面向 Unreal Engine 棕地项目的功能级工程知识图谱、可验证复用平台与 AI 信任治理层，早期服务于 5—30 人、项目已开发半年以上且希望安全引入 AI 编程的 UE C++ 团队。它位于 Codex、Claude Code、Cursor、UE MCP 与 Unreal Editor 等执行入口之下，将需求映射到 Feature、Pipeline、Implementation 与 Decision，优先复用已在明确上下文中验证的权威节点、连接和管线，并精确标出仍需人工审查的增量。

系统不以“生成更多代码”为目标，而是让程序员只审查真正新增、发生语义变化或因上下文偏差而失去验证依据的工程知识。每次审查、编译、测试与团队确认都会沉淀为可持续复用的权威资产，使未来任务的非权威增量和人工审查负担持续下降。

## Core Value

复用已经证明正确的，只审查尚未证明正确的，并让每次审查都减少未来的审查。

## Requirements

### Validated

（暂无——首个 MVP 交付并经真实审查任务验证后再晋升。）

### Active

- [ ] 固定一个明确的 UE/Lyra 版本，建立可重复运行的 MVP 基准环境。
- [ ] 人工策展“消耗体力的冲刺能力”所需的最小权威图谱，覆盖 Enhanced Input → Gameplay Ability → Gameplay Effect 管线。
- [ ] 为图谱中的节点、连接和管线记录稳定标识、实现定位、验证上下文、证据与权威状态。
- [ ] 将“增加消耗体力的冲刺能力”的需求映射到功能族，并找到可复用的权威节点与管线。
- [ ] 将候选实现分类为原样复用、配置适配、已有实现修改、新增连接或新增逻辑。
- [ ] 在执行修改前生成可解释的复用计划与权威边界。
- [ ] 在受限文件范围内生成可预览、可回滚的 Patch，并记录完整变更审计。
- [ ] 运行确定性的实体关系检查、UHT、UBT 与约定测试，并保存验证证据。
- [ ] 基于代码差异、连接变化、上下文偏差和验证结果计算待人工审查范围。
- [ ] 让审查者能够确认或拒绝非权威增量，并在确认后将新节点、连接、管线和决策写回图谱。
- [ ] 通过对照实验衡量人工审查时间、非权威增量比例、重复实现数量与缺陷遗漏情况。

### Out of Scope

- 从任意 UE 项目自动发现完整 Feature Graph 和 Pipeline Graph——首个 MVP 使用人工策展图谱，先验证权威边界的产品价值。
- 自动判断实现已经成为团队权威——权威晋升必须保留人工确认及可审计证据。
- 覆盖 Lyra 的全部功能、资产与架构——首版只覆盖一条体力冲刺垂直管线。
- 强制项目采用 Lyra 架构——Lyra 只是首个参考实现和语料基准。
- 通用多项目权威库、跨项目传播与版本兼容矩阵——待单项目闭环成立后再扩展。
- Perforce、私有 Engine 分支、企业审批、内网部署和团队权限体系——属于后续商业化或企业版本能力。
- 完整可视化知识图谱编辑器——首版围绕需求、复用计划、审查边界、影响分析和晋升流程提供最小交互面。
- 让 Agent 获得整个项目的无限写权限——所有修改必须受范围、次数、规模、预览与回滚策略约束。

## Context

当前 AI 辅助 UE 编程的主要失败并非无法生成语法正确的代码，而是缺少项目级语义、权威来源和影响上下文：它可能重复实现已有功能，无法判断当前标准实现，不理解跨 C++、Blueprint、Gameplay Ability、Gameplay Effect、Data Asset、动画、UI 与配置的功能语义，也无法证明一次修改是否仍处在既有验证条件内。结果是程序员每次都必须重新审查整个实现，上一次正确结果不能转化为长期信任资产。

系统将一次 AI 输出分为四类：已验证的权威复用、已验证的权威组合、需要重新确认的上下文适配、尚未验证的新逻辑。关键事实是“两个权威节点组成的新组合不自动成为权威”，因此权威必须分别作用于节点、连接和完整管线，并绑定明确的适用上下文。

长期数据模型由三层有向图组成：

- **Code Graph**：Module、Plugin、Class、Function、Property、Blueprint、Asset、Data Asset、Gameplay Tag、Gameplay Ability、Gameplay Effect、Config、Test 及其机械关系。
- **Feature Graph**：Feature、Feature Family 与 Pipeline，描述用户可理解的功能语义、执行顺序、条件、数据流、生命周期、客户端/服务器边界、失败路径和回滚逻辑。
- **Lineage Graph**：Implementation 的复用、派生、变体、替换与废弃关系，以及相关架构决策。

核心领域对象包括 Feature、Pipeline、Feature Family、Sister Implementation、Implementation、Decision、Authority Context、Evidence 和 Review Boundary。姊妹实现用于保留语义或生命周期差异，避免通过大量布尔参数制造万能抽象。

权威不是永久正确，而是“某实现或关系在指定上下文和验证条件下已经被证明可用”。上下文至少包括引擎版本、项目架构、插件版本、模块、平台、构建类型、网络模式、测试范围、验证提交、审查人和依赖条件。建议状态为 Candidate、Reviewed、Verified、Adopted、Authoritative、Suspect、Deprecated；UE 版本、插件、模块接口、网络模型、平台、架构或测试结果变化时，相关权威进入 Suspect 并要求重新验证。

Agent 面对需求时必须先查找功能族和权威实现，再在直接复用、配置参数化、泛化、姊妹实现和完全新增之间作出可解释选择。执行链路为：需求解析 → 功能定位 → 权威匹配 → 上下文比较 → 复用决策 → 变更计划 → 权威边界 → 隔离修改 → UHT/UBT/测试 → 影响分析 → 人工审查 → 权威晋升 → 图谱写回。

Lyra 的角色是首个权威参考项目、标准功能语料库和架构模板，而不是所有 UE 项目的唯一标准。长期系统应同时容纳 Lyra 派生、简化单机、GAS、非 GAS、网络游戏和项目自定义实现。

完整产品预计包含 `ue-agent-core`、`ue-feature-graph`、`ue-authority-registry`、`ue-project-mapper`、`ue-build-doctor`、`ue-api-verifier`、`ue-blueprint-inspector`、`ue-pipeline-tracer`、`ue-impact-analyzer` 和 `ue-agent-benchmark`。首个 MVP 不按这些长期组件一次性拆建，只实现证明核心闭环所必需的最小边界。

首个 MVP 固定一个明确 UE/Lyra 版本，并人工策展体力冲刺所需的最小权威图谱。确定性扫描器只负责验证代码实体、资产和关系仍然存在，不承担通用语义发现。该取舍将首要技术风险从“能否自动理解任意 UE 项目”收缩为“权威边界是否真实、可解释并能减少审查”。

## Constraints

- **产品验证**：首个 MVP 必须优先证明人工审查时间显著下降，同时缺陷遗漏率不升高；代码生成速度不是主要成功指标。
- **基准范围**：仅覆盖固定 UE/Lyra 版本中的 Enhanced Input → Gameplay Ability → Gameplay Effect 体力冲刺管线，以保证结果可重复。
- **知识来源**：MVP 使用人工策展的最小权威图谱；自动语义发现明确延期。
- **权威语义**：任何权威结论都必须绑定验证上下文与证据，不允许使用无条件“永久正确”标记。
- **组合规则**：节点权威、连接权威和管线权威相互独立；新组合必须重新验证。
- **确定性优先**：Engine 版本、`.uproject`、`.uplugin`、`.Build.cs`、UHT、UBT、Asset Registry、测试、Git diff 和实体关系等可确定问题不得交由 LLM 猜测。
- **安全边界**：Agent 写权限、文件数、行数、编译重试次数和可修改目录必须显式限制；默认要求 Patch 预览、审计和回滚。
- **架构中立**：Lyra 仅作为首个参考，数据模型不得把所有 UE 项目绑定为 Lyra 架构。
- **可追溯性**：复用决策、变更、验证、审查、晋升、失效和写回必须能够追溯到代码版本及证据。
- **开源优先**：本地功能图谱、权威格式、扫描器、Agent 协议、基础验证器和 Git 集成以可开源、可本地运行的形态设计。

## Success Signals

- 权威复用率可被稳定计算，且同一输入在相同代码版本和权威库上得到一致结果。
- 系统报告的非权威增量与资深 UE 程序员实际认为需要审查的范围高度一致。
- 使用系统完成体力冲刺任务的人工审查时间相较完整 Diff 审查显著下降。
- 对照审查中缺陷遗漏率不高于传统完整审查基线。
- 系统不会把新连接、生命周期变化、网络语义变化或上下文失配误报为已验证复用。
- 经人工确认的新知识能够写回，并在第二个相似任务中减少重复实现与重复审查。

## Evolution

本文档在阶段转换和里程碑边界持续演化。

**每次阶段转换后：**

1. 已失效的需求移入 Out of Scope 并记录原因。
2. 已交付并被验证的需求移入 Validated，并标注阶段。
3. 新出现且属于当前目标的需求加入 Active。
4. 将影响未来工作的关键取舍加入 Key Decisions。
5. 检查 What This Is 是否仍与实际产品一致。

**每个里程碑完成后：**

1. 复查全部章节。
2. 确认 Core Value 仍是最高优先级。
3. 审计 Out of Scope 的延期或排除理由是否仍成立。
4. 使用真实用户、反馈和指标更新 Context。
